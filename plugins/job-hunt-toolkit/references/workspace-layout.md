# Workspace Layout

The canonical directory structure for a job-hunt workspace. Default location: `~/Documents/job_seaking` (override via `JOB_HUNT_WORKSPACE`).

## Top-level structure

```
<workspace-root>/
├── <First>_<Last>_<Role>_CV.html         # master source (edit this)
├── <First>_<Last>_<Role>_CV.pdf          # master export (regenerate from HTML)
├── README.md                              # workspace overview (generated by init-workspace)
├── CLAUDE.md                              # Claude rules for the workspace (generated)
├── NAMING.md                              # naming quick reference (generated)
├── .gitignore                             # excludes .DS_Store, editor cruft
├── .git/                                  # version history
└── <company>/                             # one folder per application
```

## Master files at workspace root

The master HTML is the **canonical source**. Everything else is derived:

- **Master HTML** — edited by hand; source of truth.
- **Master PDF** — regenerated from HTML whenever HTML changes. Content must match HTML exactly.
- **Per-company HTMLs** — started as copies of master, then tailored to the JD.
- **Per-company PDFs** — regenerated from per-company HTML.

## Per-company folder

Folder name: lowercase with underscores (`acme_robotics/`, never `Acme Robotics/` or `acme-robotics/`).

### Required files

```
<company>/
├── company.md                               # company research + application status
├── job_description.(md|pdf|txt)             # raw JD
├── <First>_<Last>_<Role>_CV.html            # tailored source
└── <First>_<Last>_<Role>_CV.pdf             # tailored export (what HR receives)
```

### Optional files

```
<company>/
├── cover_letter.html
├── cover_letter.pdf
├── interview_notes.md                       # notes after each round
├── questions_to_ask.md                      # questions for interviewers
├── followups.md                             # thank-you email drafts, recruiter pings
├── salary_notes.md                          # negotiation prep (NEVER commit without user confirmation)
└── artifacts/                               # take-home exercises, coding samples
```

## `company.md` frontmatter

Keep application status in YAML frontmatter for easy grep/scanning:

```yaml
---
company: Acme Robotics
role: Senior ML Engineer
applied: 2026-04-23
status: applied | screening | interview | offer | rejected | withdrew
recruiter: name <email>
portal: https://...
comp_band: "$X–$Y"
referral: name or null
deadline: 2026-05-15 or null
priority: high | medium | low
---

## Company snapshot
- Mission:
- Stack:
- Size / stage:
- Recent news:

## Why them
- ...

## Why me (angle)
- ...

## Gaps to address
- ...
```

## Git conventions

- **Atomic commits**: HTML + PDF together, always.
- **Commit messages**:
  - `add: <company> — <role>` when adding a new application
  - `update: <company> — <reason>` when revising
  - `fix: <company> — <issue>` when correcting an error
  - `chore: <what>` for workspace-level changes
- **Never commit secrets**: salary offers, recruiter private contacts, personal IDs. If unsure, ask.
- **PDFs are binary** — git tracks them but diffs are opaque. HTML is where the real diff history lives.

## Do NOT

- Edit PDFs directly. Edit HTML, regenerate PDF.
- Put tailored versions at the workspace root.
- Put more than one company in a folder.
- Use spaces, hyphens, or mixed case in folder names.
- Commit workspace-level changes and per-company changes in the same commit (makes rollback harder).
